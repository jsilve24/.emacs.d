* Emacs maintenance plan

** Startup and load-order

*** TODO Convert `load-config` callers to feature-based ownership where practical
`bindings.el` now owns `bindings-helpers.el` directly, which is an improvement
over keeping medium-sized helper functions inline, but the broader config still
leans on raw `load` instead of explicit feature boundaries. Revisit module
pairs like `email.el`/`email-helpers.el`, `bindings.el`/`bindings-helpers.el`,
and `org-roam.el`/`org-roam-ai.el` to see where `provide`/`require` would make
startup assumptions easier to audit without increasing fragility.

*** TODO Replace raw `load` for local support modules with clearer feature ownership
`org-roam.el` loads `org-roam-ai.el` via `(load ... nil t)`, and `ai.el`
manually adds `gptel-reinforce/` to `load-path` before requiring local modules.
This works, but the ownership model is implicit and would be easier to maintain
if local package boundaries and load expectations were documented consistently.


** Module boundaries and coupling

*** TODO Re-check upstream extension points before carrying long-lived forks
Now that current fork reasons are explicit, perform a focused upstream review
for:
- framemove alternatives in EXWM built-ins
If hooks or stable APIs exist upstream, prefer dropping fork dependence.

** Public API vs internal API

*** TODO Review mu4e internals and version-sensitive assumptions
The mail setup contains comments about mu4e upgrades and behavior changes, and
it customizes deep mode behavior. Any use of version-sensitive mu4e behavior
should be documented near the workaround so future mu4e upgrades are less
fragile. Compose helpers should also avoid `message-replace-header` when the
goal is only to swap recipient values, since header rewrites can disturb the
standard `To:`/`Cc:` order in mu4e message buffers.
The junk-mark helper currently overrides `mu4e--mark-get-move-target` to feed
a deterministic target into Mu4e's move mark. Revisit that if Mu4e exposes a
public way to supply a move target without touching private internals.

*** TODO Consider replacing prompt-only scheduling confirmation heuristics with structured thread checks
`ai-email` now guards against prematurely treating proposed times as confirmed,
but that safety still depends on prompt-following. A small local checker over
the visible thread state could further reduce false-positive confirmations and
calendar staging.

*** TODO Replace the Rcpp `consult-eglot` header fallback with a public hook if upstream adds one
`eglot-rcpp/eglot-rcpp.el` now advises `consult-eglot--make-async-source` so
`SPC e e` can surface `inst/include/` symbols in plain R package workflows
where clangd does not return them from `workspace/symbol`. The behavior is
useful, but it depends on an internal async helper in `consult-eglot` and
should be revisited if the package grows a supported extension point.

** Missing guards and OS assumptions

*** TODO Review hard-coded machine names and local paths
There are several machine-specific and filesystem-specific assumptions:
- EXWM display layout keyed off exact `system-name` values
- Dropbox paths used throughout Org, references, and calendar flows
- system mu4e path in `email.el`
These may be intentional, but they should be centralized or documented so they
are easy to change later.

*** TODO Audit Slack auth and request compatibility
The Slack integration is still version-sensitive around auth, request-wrapper
handling, room lookup, and compose-buffer behavior. Keep the fork pin and local
compatibility shims only until upstream support or stable APIs make them
unnecessary.
- Recheck auth, cookie, and request-wrapper handling after Slack or request.el updates.
- Revalidate room selection, capture, attachment, and file-link helpers if emacs-slack changes its buffer properties or helper APIs.
- Remove the fork pin once the upstream fix lands.

** Consistency and ergonomics

*** TODO Document the binding convention near `bindings.el`
Global `jds/leader-def` and `jds/sub-leader-def` bindings should remain
centralized in `bindings.el` for auditability, while mode-local
`jds/localleader-def` bindings stay with their owning modules. A short comment
or maintenance note near `bindings.el` may prevent future churn around this
preference. Remember that `jds/localleader-def` already contributes the `m`
prefix, so mode-local bindings should spell only the suffix key.

*** TODO Review REPL consistency across Python, R, shell, and Stan workflows
R has extensive REPL affordances, graphics-device helpers, and localleader
bindings. Python currently has a much thinner `eval-in-repl` integration. This
is not necessarily wrong, but the differences should be intentional and
documented so future changes do not drift arbitrarily.

*** TODO Review reading-interface consistency between Mu4e and Elfeed
There is a clear attempt to align Mu4e and Elfeed navigation, which is good,
but some of it currently depends on package-specific internals and repeated key
definitions. This is a good area for future consolidation without changing the
visible workflow.

*** TODO Review Org, agenda, and capture command spread
Org-related commands and bindings are split across `org.el`, `bindings.el`,
`completing-read.el`, `snippets.el`, and `autoloads/org.el`. The workflow may
be stable, but the implementation surface is large enough that small changes
can easily break discoverability or load order.

** Documentation drift

*** TODO Add a short rationale document for `init.el` load ordering
`init.el` says load order matters and mentions `MAINTENANCE.md`, but the actual
constraints are still mostly implicit in the code. A concise explanation of
which modules must load before others would make future maintenance much safer.

*** TODO Revisit built-in C++ tree-sitter once upstream font-lock queries are fixed
Emacs 30.2's `c++-ts-mode` triggers a `treesit-query-error` during redisplay
with the current `cpp` grammar, so `treesit-auto` intentionally leaves `.cpp`
files on `c++-mode` for now. Re-enable the tree-sitter remap only after the
upstream query bug is resolved and the mode fonts cleanly in a real package
buffer.
