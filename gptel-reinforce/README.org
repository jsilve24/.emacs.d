#+title: gptel-reinforce

=gptel-reinforce= collects feedback as you work, summarizes patterns with an LLM, and incrementally improves text artifacts — AI prompts, rule sets, templates, config snippets, or any text you want to evolve from experience.

Feedback is stored locally in SQLite. No data leaves your machine except what you send to your LLM backend.

* Overview

The improvement loop works in three steps:

1. *Record feedback* while working. Call =gptel-reinforce-like=, =gptel-reinforce-dislike=, or =gptel-reinforce-score-item= whenever something good or bad happens. Each event is written immediately to SQLite.

2. *Summarize.* Call =gptel-reinforce-summarize= for an artifact. The LLM reads all feedback events since the last summary and updates a running =summary.org=. Only new events are sent each time. If review is enabled, you can inspect or edit the proposed summary before it is written.

3. *Update.* Call =gptel-reinforce-update= for an artifact. The LLM reads the summary and proposes a minimal revision to the artifact text. Review can be skipped, shown as a read-only diff, or opened as an editable =smerge= buffer. Accepted revisions are archived and become the new =current.org=.

Artifacts improve gradually, driven by real usage rather than one-shot prompting.

* Quick start

Requirements:

- Emacs 29.1 or newer, built with SQLite support
- [[https://github.com/karthink/gptel][gptel]] 0.9.9.4 or newer

If your Emacs was built without SQLite support, loading =gptel-reinforce= will fail with a clear error because feedback storage depends on the built-in =sqlite= module.

Load the package:

#+begin_src emacs-lisp
(add-to-list 'load-path (expand-file-name "gptel-reinforce" user-emacs-directory))
(require 'gptel-reinforce)
#+end_src

Register a database and then register an artifact associated with that database:

#+begin_src emacs-lisp
(gptel-reinforce-register-database
 :name "my-db"
 :candidate-fn #'my-candidate-fn)

(gptel-reinforce-register-artifact
 :name "my-artifact"
 :database "my-db"
 :type "prompt")
#+end_src

Record feedback while in a registered context:

#+begin_src emacs-lisp
(gptel-reinforce-like)              ; +1 for the current item
(gptel-reinforce-dislike)           ; -1
(gptel-reinforce-score-item nil 3)  ; arbitrary numeric score
#+end_src

Prefix any feedback command with =C-u= to attach a short note. Notes are treated as high-value evidence during summarization.

Then run the improvement loop:

#+begin_src emacs-lisp
(gptel-reinforce-summarize "my-artifact")
(gptel-reinforce-update "my-artifact")
#+end_src

* Choosing database boundaries

The most important design choice is whether two artifacts should share *item
feedback*.

Use *one database with multiple artifacts* when the artifacts operate in the
same domain and should learn from the same notion of /what matters/. In one
database:

- item-feedback is shared across all artifacts
- output-feedback stays scoped to the artifact that produced the output

Example: an email triage assistant might include a classifier, a summary
artifact, and a reply-drafting artifact. You may want item feedback like "this
thread mattered" to help the classifier and the summary prompt, while still
keeping output feedback about a bad generated reply attached only to the reply
artifact.

Use *separate databases* when the artifacts should not share even domain-level
item signal.

Example: if you have one artifact for scheduling replies and another for email
summarization, "this thread mattered" may be irrelevant or actively misleading
when revising the scheduling-reply prompt. In that setup, two databases are
usually cleaner: one for reply-generation artifacts and one for summarization
artifacts.

Rule of thumb:

- Use item feedback when the artifact is learning what to notice, rank, select,
  or prioritize in the underlying item.
- Use output feedback when the artifact is learning how to phrase or structure
  generated text.
- If you mostly care about prompt quality and do not want item relevance to
  influence updates, either use separate databases or configure your workflow so
  you rely almost entirely on output feedback.

For the predefined Elfeed integration ([[*Elfeed (elfeed-score integration)][details]]), just call:

#+begin_src emacs-lisp
(gptel-reinforce-register-elfeed-module)
#+end_src

* Core concepts

** Databases

A *database* represents one application context — an RSS reader, email client, file navigator, etc. It owns:

- A SQLite file at =<state-root>/<name>.sqlite= for all feedback events
- A root directory for artifact org files
- A *candidate function* that says whether this database applies right now and, if so, describes the current item

Multiple artifacts can share one database.

** Artifacts

An *artifact* is a piece of text you want to improve — a prompt, rule set, template, or config file. It belongs to one database and lives in three files on disk:

- =current.org= — the active version of the text and metadata
- =summary.org= — the running feedback summary and =LAST_EVENT_ID= (used to prevent reprocessing already summarized feedback)
- =history/<artifact>/<timestamp>-<name>.org= — immutable snapshots of every accepted version

See [[*Appendix B: Org file formats][Appendix B]] for the file formats.

You can edit =current.org= directly at any time. The =* Current Text= heading holds the artifact body. The =* Summarizer User Prompt= and =* Updater User Prompt= headings let you add task-specific guidance that is appended to the default system prompts on every LLM call to summarise the feedback database or update the artifact based on the summary respectively.

** Feedback types

There are two kinds of feedback events. The difference is /what you are reacting to/.

*Item feedback* — you are reacting to the item itself (an RSS entry, an email, a file). Call =gptel-reinforce-like= or =gptel-reinforce-dislike= while viewing the item. For example, thumbs-up on an elfeed entry means "I want to see more entries like this." This signal is *database-scoped*: every artifact in the database sees it when summarizing, because the signal is about the domain, not about any one artifact.

*Output feedback* — you are reacting to AI-generated text that a specific artifact version produced. For example, if an artifact generated a triage summary and the summary was wrong, you dislike /that output/. This signal is *artifact-scoped*: the event is stored with the producing artifact's name, and when =gptel-reinforce-summarize= runs it only includes output-feedback events that name that specific artifact — other artifacts in the same database never see it.

Output feedback requires manual wiring. There is no automatic hook into gptel. Inside your LLM request callback, after inserting the response text, call =gptel-reinforce-track-output-region= to annotate the inserted region:

#+begin_src emacs-lisp
(gptel-request my-prompt
  :callback
  (lambda (response _info)
    (let ((start (point)))
      (insert response)
      (gptel-reinforce-track-output-region "my-artifact" start (point)))))
#+end_src

After that, =gptel-reinforce-like= and friends at that region automatically record output-feedback. Outside that region they record item-feedback as normal.

For the predefined elfeed integration you only ever use item feedback and none of this applies. Output feedback is only relevant if you build a custom workflow where an artifact generates text displayed to the user (i.e., LLM text completion).

** Candidate functions

A candidate function is called in the current buffer on every item-feedback command. It must return either nil or a candidate plist. Returning nil means "this database does not apply here right now."

The candidate plist must contain =:context=. It may also include =:priority= and =:label=.

| Candidate key | Type    | Description                                                   |
|---------------+---------+---------------------------------------------------------------|
| =:context=    | plist   | *Required.* Item context used when recording feedback         |
| =:priority=   | integer | Optional. Higher numbers win when multiple databases match    |
| =:label=      | string  | Optional. Human-readable name shown in the chooser            |

The =:context= plist must contain =:item-key=. All other context keys are optional:

| Key             | Type   | Description                                  |
|-----------------+--------+----------------------------------------------|
| =:item-key=     | string | *Required.* Stable unique ID for this item   |
| =:title=        | string | Human-readable label shown in messages       |
| =:primary-text= | string | Main content — excerpt, snippet, body        |
| =:meta=         | plist  | Additional structured metadata (any keys)    |

When several databases return candidates, =gptel-reinforce= chooses the one with the highest =:priority=. If priorities tie, it can prompt you to choose. The chooser uses =:label= when present.

* Commands

** Feedback commands

All feedback commands accept =C-u= as a prefix to prompt for an optional note.

The base commands (=gptel-reinforce-like=, =-dislike=, =-neutral=, =-score=) auto-detect context: if point is inside annotated LLM output they record output-feedback against that artifact; otherwise they ask the registered databases for item candidates and use the highest-priority match. These are the ones you typically bind to keys — they do the right thing in both contexts.

The =-item= variants (=gptel-reinforce-like-item=, etc.) always record item-feedback regardless of where point is. Use them when you are inside annotated output but want to rate the underlying item rather than the output. =gptel-reinforce-score-output= always records output-feedback. It looks for provenance first at point (text properties), then falls back to the most recently tracked output in the current buffer (=gptel-reinforce-last-output-provenance=). It errors only if neither exists — meaning =gptel-reinforce-track-output-region= was never called in this buffer, so there is genuinely no record of which artifact or version produced the output.

| Command                         | Score | Target                             |
|---------------------------------+-------+------------------------------------|
| =gptel-reinforce-like=          | +1    | Output at point, else current item |
| =gptel-reinforce-dislike=       | -1    | Output at point, else current item |
| =gptel-reinforce-neutral=       | 0     | Output at point, else current item |
| =gptel-reinforce-score=         | N     | Output at point, else current item |
| =gptel-reinforce-like-item=     | +1    | Current item only                  |
| =gptel-reinforce-dislike-item=  | -1    | Current item only                  |
| =gptel-reinforce-neutral-item=  | 0     | Current item only                  |
| =gptel-reinforce-score-item=    | N     | Current item only                  |
| =gptel-reinforce-score-output=  | N     | Output at point only               |

** Summarize and update

*=gptel-reinforce-summarize ARTIFACT=* — reads all feedback events for ARTIFACT since =LAST_EVENT_ID= and asks the LLM to update =summary.org=. Only new events are sent; summarization is incremental and safe to run repeatedly. Review behavior is controlled by =gptel-reinforce-summary-review-mode=.

*=gptel-reinforce-update ARTIFACT=* — sends the current artifact text and =summary.org= to the LLM and requests a minimal revision. Review behavior is controlled by =gptel-reinforce-update-review-mode= and can be skipped per artifact with =AUTO_UPDATE: t= in =current.org=. Accepted revisions are written to the history directory and become the new =current.org=.

** Utility

*=gptel-reinforce-set-active-database DATABASE=* — pins a database to the current buffer. Feedback commands normally auto-detect by asking every registered database for a candidate, then choosing the highest-priority match. If you want a buffer-local override regardless of candidate priority, pin the database explicitly. The typical use is in a mode hook:

#+begin_src emacs-lisp
(add-hook 'my-special-mode-hook
          (lambda ()
            (gptel-reinforce-set-active-database "my-db")))
#+end_src

* Predefined integrations

** Elfeed (elfeed-score integration)

Tracks your thumbs-up/thumbs-down feedback on RSS entries and uses it to iteratively improve your [[https://github.com/sp1ff/elfeed-score][elfeed-score]] rule file (=elfeed.score=). Each time you run =gptel-reinforce-update=, the LLM receives a summary of your recent feedback and proposes a revised score file; if you accept the diff, the file is written to disk and reloaded immediately.

#+begin_src emacs-lisp
(gptel-reinforce-register-elfeed-module)
#+end_src

*Seeding*: on first call, if =elfeed.score= already exists and the artifact is empty, the file is read in automatically so the LLM has a baseline to work from.

*Validation*: before any LLM-proposed score file is accepted, a pre-update hook checks that it parses as valid Lisp. Invalid output is rejected without touching your score file.

*Reload*: after an accepted update, the post-update hook writes the new text to =gptel-reinforce-elfeed-score-file= and calls =elfeed-score-load-score-file=, so scoring takes effect immediately without restarting Emacs.

The Elfeed candidate function works in both =elfeed-search-mode= and =elfeed-show-mode=, so you can give feedback from the search list or from inside an entry.

To re-seed the artifact from the score file after editing it manually:

#+begin_src emacs-lisp
(gptel-reinforce-elfeed-seed-score-file 'force)
#+end_src

* Writing your own integration

** 1. Write a candidate function

The candidate function is called in the user's current buffer on every item-feedback command. Return nil when this database does not apply. Return a candidate plist when it does.

#+begin_src emacs-lisp
(defun my-candidate ()
  "Return a candidate for my-mode, or nil when not applicable."
  (when (derived-mode-p 'my-mode)
    (when-let* ((item (my-current-item)))
      (list :priority 10
            :label "My mode"
            :context
            (list :item-key     (my-item-id item)     ; required — must be stable
                  :title        (my-item-title item)  ; shown in echo area
                  :primary-text (my-item-body item)   ; sent to the LLM
                  :meta         (list :source (my-item-source item)))))))
#+end_src

Guard with =(featurep ...)= and =(derived-mode-p ...)= so the function returns nil cleanly in unrelated buffers. =:item-key= should be a stable identifier that doesn't change across sessions — a URL, message-id, hash, etc. Use =:priority= only when multiple databases may plausibly apply in the same buffer and one should win by default.

** 2. Register the database and artifact

#+begin_src emacs-lisp
(gptel-reinforce-register-database
 :name "my-db"             ; required — unique string key
 :candidate-fn #'my-candidate) ; required

(gptel-reinforce-register-artifact
 :name "my-artifact"       ; required — unique within the database
 :database "my-db"         ; required
 :type "prompt"            ; optional — informational ("prompt", "code", "rules", …)
 :auto-update nil          ; if t, skip the update review step entirely
 :summarizer-user-prompt "Focus especially on X and Y."
 :updater-user-prompt     "Prefer conservative edits.")
#+end_src

See the full key reference in [[*Appendix C: Configuration variables][Appendix C]].

** 3. Bind feedback commands

#+begin_src emacs-lisp
(define-key my-mode-map (kbd "C-c +") #'gptel-reinforce-like)
(define-key my-mode-map (kbd "C-c -") #'gptel-reinforce-dislike)
#+end_src

Or set the active database in a mode hook to override candidate-based selection entirely:

#+begin_src emacs-lisp
(add-hook 'my-mode-hook
          (lambda () (gptel-reinforce-set-active-database "my-db")))
#+end_src

** Using hooks

Hooks let you enforce constraints on LLM output before it is saved and propagate accepted changes to their destination after saving. They are the mechanism that makes an artifact /live/ — without them you would need to manually copy text out of =current.org= and apply it yourself.

*Pre-update hooks* run before the new version is written. Each hook receives =(artifact current-record candidate-text)= and must return non-nil to accept or nil to reject the candidate. If any hook returns nil, the update is aborted and the artifact is left unchanged.

The primary use is *format validation*: LLMs occasionally produce malformed output, and writing bad data to a config file or prompt can silently break things. A pre-update hook is your safety check:

#+begin_src emacs-lisp
(defun my-validate (_artifact _current-record candidate)
  "Return non-nil when CANDIDATE is valid Lisp."
  (condition-case nil
      (progn (read-from-string candidate) t)
    (error nil)))
#+end_src

You can also use a pre-update hook to *compare against the existing version* — for example, to reject a candidate that is shorter than the current text by more than a threshold, or that drops lines matching a required pattern.

*Post-update hooks* run after the new version is written to the org file. Each hook receives =(artifact version-ref current-record candidate-text)=. They are for *side-effects that apply the artifact to its destination* — without a post-update hook, updates accumulate in the org file but never take effect anywhere.

Typical uses:
- Write the accepted text to a config file and reload it (elfeed-score, a shell alias file, a CSS theme, etc.)
- Copy the text to the clipboard or a kill-ring entry for manual pasting
- Call a reload command (=load-file=, =elfeed-score-load-score-file=, =css-reload=, …)
- Send a notification that the artifact was updated

#+begin_src emacs-lisp
(defun my-apply (_artifact _version-ref _current-record candidate)
  "Write CANDIDATE to the config file and reload it."
  (with-temp-file my-config-file
    (insert candidate))
  (my-reload-config))
#+end_src

Register hooks at artifact creation time:

#+begin_src emacs-lisp
(gptel-reinforce-register-artifact
 :name "my-artifact"
 :database "my-db"
 :pre-update-hook  #'my-validate   ; or a list of functions
 :post-update-hook #'my-apply)
#+end_src

Both keys accept either a single function or a list of functions. For pre-update hooks, all functions in the list must return non-nil for the update to proceed.

** Tracking output provenance

To capture output-feedback that traces back to the exact artifact version that produced it, annotate the inserted text:

#+begin_src emacs-lisp
(let ((start (point)))
  (insert response)
  (gptel-reinforce-track-output-region "my-artifact" start (point)))
#+end_src

After annotation, =gptel-reinforce-like= and friends at that text automatically record output-feedback rather than item-feedback.

For text not inserted into a buffer, use =gptel-reinforce-propertize-output= to get an annotated string. A common case is displaying LLM output in a transient pop-up, an overlay, or a tooltip — places where the text is rendered from a string variable rather than inserted into a buffer that the user navigates. Another example is a completing-read list where each candidate is an LLM-generated summary: the candidate strings carry provenance as text properties so that selecting and scoring one records output-feedback against the right artifact version. In both cases the pop-up can disappear; as long as you hold the annotated string (or set =gptel-reinforce-last-output-provenance= before scoring) the feedback will be recorded against the right artifact version.

#+begin_src emacs-lisp
;; Example: annotate a completing-read candidate list
(let* ((raw-summaries (my-generate-summaries artifact-name))
       (annotated (mapcar (lambda (s)
                            (gptel-reinforce-propertize-output artifact-name s))
                          raw-summaries))
       (chosen (completing-read "Pick a summary: " annotated)))
  ;; Point is now "on" the chosen string; score-output will find its provenance.
  (gptel-reinforce-like nil))
#+end_src

* Appendix A: Database schema

Each database has one SQLite file. The only table is =feedback_events=. Rows are never updated or deleted — the table is append-only.

** =feedback_events=

| Column                 | Type    | Null | Description                                                   |
|------------------------+---------+------+---------------------------------------------------------------|
| =id=                   | INTEGER | no   | Primary key, monotonically increasing                         |
| =created_at=           | TEXT    | no   | ISO-8601 timestamp                                            |
| =event_type=           | TEXT    | no   | ="item-feedback"= or ="output-feedback"=                      |
| =item_key=             | TEXT    | yes  | Stable identifier from the candidate context                  |
| =score=                | REAL    | no   | Numeric score (−1 / 0 / +1 or any float)                     |
| =title=                | TEXT    | yes  | Human-readable label at feedback time                         |
| =primary_text=         | TEXT    | yes  | Content excerpt at feedback time                              |
| =meta_json=            | TEXT    | yes  | JSON-encoded =:meta= plist from the candidate context         |
| =note=                 | TEXT    | yes  | Optional free-text note attached by the user                  |
| =artifact_name=        | TEXT    | yes  | Non-null only for ="output-feedback"= events                  |
| =artifact_version_ref= | TEXT    | yes  | History filename of the artifact version that produced output |
| =output_id=            | TEXT    | yes  | Opaque ID linking an event to a specific output region        |

** Indexes

- =feedback_events_event_type_idx= on =event_type=
- =feedback_events_artifact_name_idx= on =artifact_name=

** Summarization query

When summarizing artifact =A= in database =D= with last-event-id =N=, the query is:

#+begin_src sql
SELECT ... FROM feedback_events
WHERE id > N
  AND (event_type = 'item-feedback'
       OR (event_type = 'output-feedback' AND artifact_name = 'A'))
ORDER BY id ASC
#+end_src

All item-feedback in the database is visible to every artifact in that database. Output-feedback is scoped to the specific artifact.

* Appendix B: Org file formats

** =current.org=

Stores the active artifact text and metadata. Edit this file directly.

#+begin_example
:PROPERTIES:
:DATABASE:    <database-name>
:ARTIFACT:    <artifact-name>
:TYPE:        <type string, e.g. "prompt" or "code">
:VERSION_REF: <history filename for this version>
:UPDATED_AT:  <ISO-8601 timestamp>
:AUTO_UPDATE: t | nil
:END:

* Current Text
 <artifact text — each line indented by one space>

* Applied Summary
 <snapshot of the feedback summary that was used in the last accepted update>

* Summarizer User Prompt
 <optional guidance appended to the summarizer system prompt>

* Updater User Prompt
 <optional guidance appended to the updater system prompt>
#+end_example

Set =AUTO_UPDATE: t= to skip the update review step during =gptel-reinforce-update=, regardless of =gptel-reinforce-update-review-mode=.

=Applied Summary= is written automatically whenever an update is accepted. It holds a snapshot of =summary.org= at the moment of that acceptance. On the next =gptel-reinforce-update= call, the LLM sees both this snapshot ("already incorporated") and the current summary ("new since last update"), so it only acts on the delta. If you reject an update, =Applied Summary= is not changed — the next attempt still sees everything since the last accepted update.

=VERSION_REF= is the basename of the history file for this version, e.g. =20240915T142300-my-artifact.org=. It is used to link output-feedback events back to the exact version.

** =summary.org=

Stores the running feedback summary. Updated by =gptel-reinforce-summarize=.

#+begin_example
:PROPERTIES:
:DATABASE:      <database-name>
:ARTIFACT:      <artifact-name>
:LAST_EVENT_ID: <integer>
:UPDATED_AT:    <ISO-8601 timestamp>
:END:

* Summary
<LLM-maintained summary of stable feedback patterns>

* Uncertainty
<Mixed or weak evidence>

* Notes
<High-value observations from user notes>
#+end_example

=LAST_EVENT_ID= is the =id= of the most recent event included in this summary. Summarization reads only events with =id > LAST_EVENT_ID=, so it is always incremental.

** History snapshots

Each accepted update creates a file at =history/<artifact>/<timestamp>-<name>.org=:

#+begin_example
:PROPERTIES:
:DATABASE:                     <database-name>
:ARTIFACT:                     <artifact-name>
:TYPE:                         <type>
:UPDATED_AT:                   <ISO-8601 timestamp>
:UPDATE_MODE:                  manual-approved | auto-updated | initial
:SOURCE_SUMMARY_LAST_EVENT_ID: <last-event-id at time of update>
:END:

* Current Text
 <artifact text at this version>
#+end_example

The filename (without directory) becomes =VERSION_REF= in =current.org= and =artifact_version_ref= in the database, linking feedback events to the exact version of the artifact that produced them.

| Variable                               | Default   | Description                                                      |

** LLM backend

| Variable                                    | Default                                   | Description                        |
|---------------------------------------------+-------------------------------------------+------------------------------------|
| =gptel-reinforce-backend-function=          | =gptel-reinforce-backend-send-with-gptel= | Function used to send LLM requests |
| =gptel-reinforce-default-summarizer-prompt= | (see source)                              | System prompt for summarization    |
| =gptel-reinforce-default-updater-prompt=    | (see source)                              | System prompt for artifact updates |

To swap the LLM transport entirely, set =gptel-reinforce-backend-function= to a function with signature =(request callback)=. =request= is either a =gptel-reinforce-summary-request= or =gptel-reinforce-update-request= struct; =callback= is =(lambda (response info) ...)=.

#+begin_src emacs-lisp
(setq gptel-reinforce-backend-function
      (lambda (request callback)
        ;; Build prompt from request, call your LLM, then:
        (funcall callback "revised artifact text" nil)))
#+end_src

| =:pre-update-hook=           | no       | nil                                         | Function or list; return nil to reject candidate        |
| =:post-update-hook=          | no       | nil                                         | Function or list; called after acceptance               |

* Appendix D: LLM prompt formats and data interception

** Summarization prompt

The summarization user prompt is plain text built from a =gptel-reinforce-summary-request= struct. It looks like this:

#+begin_example
Database: elfeed-ranking
Artifact: elfeed-score-rules
Artifact type: code

Existing summary:
<current contents of summary.org * Summary heading>

New feedback events:
- id: 42
  created_at: 2026-04-10T14:32:00+0000
  event_type: item-feedback
  score: 1
  title: Efficient Transformers: A Survey
  primary_text: We present a survey of efficient Transformer models...
  meta: {"feed_title":"ArXiv cs.LG","tags":["unread"]}

- id: 43
  created_at: 2026-04-10T14:33:10+0000
  event_type: item-feedback
  score: -1
  title: Sponsored: Best VPN deals
  primary_text: ...

Update the summary conservatively and keep it useful for future edits.
#+end_example

Optional fields (=item_key=, =artifact_name=, =artifact_version_ref=, =output_id=, =note=) are included per-event only when non-nil. The system prompt is =gptel-reinforce-default-summarizer-prompt= plus any task-specific guidance appended from =:summarizer-user-prompt=.

** Update prompt

#+begin_example
Database: elfeed-ranking
Artifact: elfeed-score-rules
Artifact type: code

Current artifact text:
<current contents of current.org * Current Text heading>

Feedback already incorporated in current version:
<current contents of current.org * Applied Summary heading>
(shows "(none — first update)" when empty)

Latest feedback summary (may include new patterns since last update):
<current contents of summary.org * Summary heading>

Revise the artifact with the smallest useful change to reflect
any new patterns in the latest summary not yet incorporated.
Do not re-apply changes already present in the artifact.
#+end_example

The "already incorporated" section is the snapshot saved when the previous update was accepted. The "latest summary" is the full current =summary.org= body, which may have grown since then. The LLM can diff the two mentally and act only on the delta. If no update has ever been accepted, the incorporated section is empty and the full summary is treated as new.

The system prompt is =gptel-reinforce-default-updater-prompt= plus any task-specific guidance from =:updater-user-prompt=.

** Interception points

There are two ways to bypass or augment the LLM pipeline.

*Replace the backend function* — set =gptel-reinforce-backend-function= to your own =(request callback)= function. You receive the fully-populated request struct before any prompt is built. Access the raw event list directly:

#+begin_src emacs-lisp
(setq gptel-reinforce-backend-function
      (lambda (request callback)
        (if (gptel-reinforce-summary-request-p request)
            (let ((events (gptel-reinforce-summary-request-events request)))
              ;; events is a list of plists: (:id N :score N :title "..." :primary-text "..." ...)
              ;; pass to a classifier, write results somewhere, then call callback:
              (funcall callback (my-classifier-summarize events) nil))
          ;; fall through to default for update requests
          (gptel-reinforce-backend-send-with-gptel request callback))))
#+end_src

*Query the database directly* — =gptel-reinforce-db-feedback-since= returns raw event plists from SQLite without going through the summarize/update pipeline at all. Useful for batch processing, export, or feeding a classifier on a schedule independently of artifact management:

#+begin_src emacs-lisp
(let* ((db (gptel-reinforce-resolve-database "elfeed-ranking"))
       (events (gptel-reinforce-db-feedback-since db 0)))
  ;; events: list of plists with :id :score :title :primary-text :meta ...
  (dolist (event events)
    (my-spam-classifier (plist-get event :primary-text)
                        (plist-get event :score))))
#+end_src

Pass =0= as the second argument to get all events ever recorded. Pass a specific event id to get only events after that point (the same mechanism =gptel-reinforce-summarize= uses to avoid reprocessing). Pass an artifact name as the optional third argument to filter to output-feedback for that artifact only.
